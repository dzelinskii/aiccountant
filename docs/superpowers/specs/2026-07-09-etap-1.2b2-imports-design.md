# AIccountant — этап 1.2b2 «Импорт выписки Т-Банка (PDF)»

Дата: 2026-07-09. Статус: на ревью.

Под-проект этапа 1 (ledger), часть B, шаг 2. Идёт после B1 (Celery/beat +
регулярка, смержено) и опциональной категории (смержено). Общая спека:
`docs/superpowers/specs/2026-07-02-aiccountant-design.md`.

## 1. Цель

Импорт банковской выписки в приложение, чтобы не вводить операции руками — в
духе курса на автоматизацию. Реальный источник у пользователя — **PDF «Справка
о движении средств» Т-Банка** (CSV банк не отдаёт). Парсер детерминированный,
под фиксированный формат этой справки. Разбор произвольных форматов и других
банков (LLM) — этап 2.

## 2. Скоуп

Включено:

- Новый модуль `app/imports/`: разбор PDF Т-Банка, превью, подтверждение,
  вставка операций.
- Детерминированный парсер формата «Справка о движении средств»: строки
  операций → знаковые транзакции; профиль `tbank_statement`.
- Дедупликация по `external_id` (хеш), уникальный индекс
  `(account_id, external_id)`; повторный импорт того же файла не задваивает.
- Синхронный поток превью → подтверждение (без Celery).
- Импортированные операции: `source=import`, категория пустая (разгребёт AI
  на этапе 2), знак из выписки.
- Frontend: страница «Импорт» (выбор счёта, загрузка PDF, превью, подтверждение).
- Миграция `0005`: таблица `imports`, колонки `external_id`/`import_id` в
  `transactions`.
- import-linter: контракт границ для `imports`.

Явно исключено (этап 2 или позже):

- Другие банки и произвольные форматы; LLM/OCR-разбор сканов.
- Асинхронный импорт через Celery (станет честным, когда добавится
  медленный OCR/LLM-разбор — тогда вынесем в фон, пометив осознанно).
- Редактирование строк в превью; откат импорта; автокатегоризация.
- Парный `transfer_group_id` — каждая строка выписки заводится как обычная
  знаковая операция (у нас только одна сторона движения).

## 3. Формат «Справки о движении средств» (наблюдения по реальному файлу)

- PDF с текстовым слоем (не скан). Шапка: реквизиты банка, ФИО, договор,
  лицевой счёт, период. Затем таблица операций, в конце — итоги и подпись.
- Колонки: «Дата и время операции», «Дата списания», «Сумма в валюте
  операции», «Сумма операции в валюте карты», «Описание операции»,
  «Номер карты».
- Сумма — **знаковая** (`-1 150.00 ₽` / `+5 000.00 ₽`), разделитель тысяч —
  пробел, десятичный — точка, символ `₽`. У рублёвого счёта обе суммы равны.
- Дата операции — `ДД.ММ.ГГГГ` + время `ЧЧ:ММ`. Описание бывает многострочным
  (переносы). Номер карты — последние 4 цифры.
- **Уникального id операции в PDF нет.**
- Футер: строки `Пополнения: <сумма> ₽` и `Расходы: <сумма> ₽` (здесь
  десятичный разделитель — **запятая**), затем подпись/реквизиты — всё это
  парсер игнорирует, а суммы итогов использует как контрольную проверку.

## 4. Парсер

Двухслойно ради тестируемости и изоляции хрупкой части:

- `imports/parser.py::extract_lines(pdf_bytes) -> list[str]` — тонкий слой:
  извлекает текст/строки из PDF (библиотека **pdfplumber**; ставится в
  backend). Здесь вся зависимость от PDF и layout.
- `imports/parser.py::parse_statement(lines) -> ParsedStatement` — **чистая
  функция** (без БД и PDF): группирует строки в операции, разбирает дату,
  знаковую сумму, описание; собирает итоги-контрольную сумму. Именно она
  покрывается юнит-тестами на текстовых фикстурах.

`ParsedStatement`: `operations: list[ParsedOperation]`, `total_income: Decimal`,
`total_expense: Decimal`. `ParsedOperation`: `occurred_at: date`,
`amount: Decimal` (знаковая), `currency: str` ("RUB"), `description: str`.

Правила разбора:

- `occurred_at` — дата из «Дата и время операции» (время отбрасываем, храним
  дату; поле `occurred_at` в модели — `date`).
- `amount` — «Сумма операции в валюте карты», знак сохраняется; парсинг:
  убрать пробелы-разделители и `₽`, точка — десятичный → `Decimal`.
- `description` — склейка многострочного описания в одну строку (нормализация
  пробелов), пойдёт в `merchant` транзакции (обрезка до 300).
- Контроль: `sum(amount>0)` ≈ `total_income`, `sum(-amount<0)` ≈
  `total_expense`; расхождение → предупреждение в structlog (не отказ).
- Некорректный/непарсящийся PDF (нет ожидаемой шапки/колонок) → доменная
  ошибка `StatementParseError` → 422 с понятным сообщением.

## 5. Данные (миграция `0005`)

- `imports`:
  - `id` uuid PK
  - `workspace_id` uuid FK → workspaces
  - `account_id` uuid FK → accounts
  - `file_name` str(300)
  - `bank_profile` str(30) — пока всегда `tbank_statement`
  - `status` str(20) — `completed` (в синхронном потоке запись создаётся уже
    завершённой; поле оставляем под будущий async)
  - `stats` JSONB — `{parsed, imported, duplicates}`
  - `created_by` uuid FK → users
  - `created_at` timestamptz
- `transactions` (таблица ledger) получают:
  - `external_id` str(64) nullable — дедуп-хеш;
  - `import_id` uuid nullable — мягкая ссылка на `imports.id` (без FK, чтобы
    ledger не зависел от imports).
  - Уникальный индекс `uq_transactions_account_external`
    `(account_id, external_id)` частичный `WHERE external_id IS NOT NULL`.
- `external_id` = sha256-хеш строки
  `f"{account_id}|{occurred_at}|{amount}|{description}"` (hex). Детерминирован
  → повторный импорт даёт те же ключи → дедуп.

Модель `Transaction` (ledger) расширяется полями `external_id`/`import_id`
(nullable). Это единственная правка ledger; постинг остаётся через
`ledger.service.post_transaction`, а `external_id`/`import_id` проставляются на
возвращённом (ещё не закоммиченном) объекте транзакции внутри `imports.service`.

## 6. API

Модуль `imports`, роутер под `/api`. Требует `require_workspace_member`;
`workspace_id`, `account_id` — query-параметры; счёт должен принадлежать
workspace (иначе 403/404).

- `POST /api/imports?workspace_id=&account_id=&commit=false` — multipart
  (`file`: PDF). Разобрать, сверить с уже импортированными по
  `(account_id, external_id)`; вернуть превью:
  ```
  {
    "operations": [{occurred_at, amount, currency, description, is_duplicate}],
    "new_count": int, "duplicate_count": int,
    "total_income": str, "total_expense": str
  }
  ```
  Ничего не пишет.
- `POST /api/imports?workspace_id=&account_id=&commit=true` — тот же multipart;
  парсит заново (не доверяя клиенту), создаёт запись `imports`, вставляет
  только новые операции (`source=import`, `category_id=null`, `external_id`,
  `import_id`), возвращает `{import_id, imported, duplicates}`. Дубли
  пропускаются молча. Атомарно (один commit).

Ограничение размера файла и content-type (`application/pdf`) проверяются;
превышение/неверный тип → 422/413.

## 7. Frontend

- Пункт «Импорт» в навигации AppShell.
- Страница: `Select` счёта + `FileInput` (PDF) → кнопка «Разобрать» (POST
  commit=false) → таблица превью (дата, сумма строкой, описание, бейдж
  «новая»/«дубль») + счётчики (новых/дублей) и итоги → кнопка «Импортировать
  N новых» (POST commit=true, повторная отправка того же файла) → результат
  (сколько импортировано) и обновление лент/дашборда.
- Суммы — строками; знак из выписки; `Intl.NumberFormat`.

## 8. Тестирование

- Backend (pytest):
  - **`parse_statement`** на текстовых фикстурах: знаковые суммы (пробел-
    разделитель, точка-десятичный), многострочные описания, пропуск футера/
    итогов, контрольная сумма, пустая/битая выписка → `StatementParseError`;
  - **дедуп**: импорт файла → N новых; повторный импорт того же → 0 новых,
    N дублей; `external_id` детерминирован;
  - импортированные операции: `source=import`, `category_id=null`, попадают в
    ленту и в бакет «Без категории» на дашборде;
  - **изоляция workspace**: импорт в чужой счёт/ workspace → 403/404;
  - интеграционный тест полного разбора — на **синтетическом** PDF (генерится
    в тесте, напр. reportlab-dev-зависимость, воспроизводя layout; без PII).
    Реальный файл пользователя в репозиторий не кладётся.
- Frontend (vitest): превью-таблица и счётчики на замоканном API; кнопка
  импорта шлёт `commit=true`.
- import-linter: контракт `imports` (не лезет в `identity`/`ledger.repository`/
  `ledger.models`, только `ledger.service`), проверяется пробой.

## 9. Принятые развилки

- Источник — PDF (CSV банк не отдаёт); парсер детерминированный под формат
  Т-Банка; произвольные форматы/LLM — этап 2.
- Дедуп по хешу `(account_id, дата, сумма, описание)` — id операции в PDF нет.
- Синхронный поток превью→подтверждение; Celery для импорта отложен до
  появления медленного OCR/LLM-разбора (тогда — осознанно в фон).
- Импортированные операции без категории (`source=import`) — вход для будущей
  AI-категоризации; направление из знака суммы.
- `import_id` — мягкая ссылка (uuid без FK), чтобы ledger не зависел от imports.
- Постинг только через `ledger.service`; `imports` не трогает таблицы ledger.

## 10. Открытые вопросы

- Порог размера файла и постраничные разрывы длинных выписок — параметры
  парсера уточняются на реальных многостраничных файлах (текущий — 4 страницы).
- Сопоставление номера карты выписки со счётом приложения не делаем (счёт
  выбирает пользователь); при мультикартовых счетах вернёмся при необходимости.
- Формат итогов с запятой-десятичным (в отличие от строк операций с точкой)
  учитывается только для контрольной суммы.
