# AIccountant — этап 1.2b1 «Celery/beat + регулярные платежи»

Дата: 2026-07-08. Статус: на ревью.

Под-проект этапа 1 (ledger), часть B, шаг 1. Общая спека продукта:
`docs/superpowers/specs/2026-07-02-aiccountant-design.md`. Ядро учёта (часть A):
`docs/superpowers/specs/2026-07-03-etap-1.2a-ledger-core-design.md` (смержено,
PR #4). Часть B шаг 2 (импорт выписки Т-Банка) — отдельная спека, после B1.

## 1. Цель

Регулярные платежи домохозяйства: правило с расписанием «каждые N периодов»
порождает срабатывания, которые либо **автоматически проводятся** транзакцией
(`autopost`), либо ждут ручного подтверждения в in-app списке «к оплате»
(`remind`). Заодно этап честно вводит фоновую инфраструктуру — Celery-воркер и
beat, которые до этого в проекте не использовались.

Строится поверх identity (1.1) и ledger (1.2a). Уведомления (Telegram/почта) —
позже; в B1 `remind` живёт только внутри приложения.

## 2. Скоуп

Включено:

- Celery + beat как отдельные процессы того же Docker-образа; брокер и
  result backend — существующий Redis. Периодический скан правил.
- Модуль `recurring`: правила (`recurring_rules`) и срабатывания
  (`recurring_occurrences`).
- Расписание «каждые N <период>» (день/неделя/месяц/год) с якорем и датой
  старта; вычисление `next_run_at` вперёд.
- Режимы `autopost` (проведение транзакции) и `remind` (пункт в списке
  «предстоящие/просроченные» с подтверждением/пропуском).
- Автопроведение и подтверждение создают транзакции `source=recurring`
  **через публичный сервис ledger**, не трогая его таблицы напрямую.
- Frontend: страница «Регулярные платежи» — CRUD правил + блок «К оплате».
- import-linter: новый контракт границ для модуля `recurring`.

Явно исключено (позже):

- Уведомления Telegram/почтой (модуль `notifications`); в B1 `remind` —
  только in-app список.
- Импорт выписок, дедупликация, колонки `external_id`/`import_id` (часть B
  шаг 2).
- Переводы в регулярке (правило = один счёт + одна категория; только
  расход/доход).
- Бэкфилл пропущенных периодов (сознательно — см. §7).
- Счётчик числа срабатываний как условие остановки; сложные расписания
  (cron/RRULE), несколько категорий в одном правиле.

## 3. Модель данных

Все таблицы несут `workspace_id`. Деньги — `NUMERIC(20, 4)`/`Decimal`.
Миграция `0003_recurring` через `alembic revision --autogenerate` с ручной
вычиткой имён/порядка.

### recurring_rules

- `id` uuid PK
- `workspace_id` uuid FK → workspaces
- `account_id` uuid FK → accounts
- `category_id` uuid FK → categories
- `amount` NUMERIC(20,4) — **знаковая**; знак сверяется с `category.kind`
  (расход < 0, доход > 0), как у транзакций
- `currency` str(3) — денормализовано со счёта на момент создания
- `period` str(10) — day / week / month / year
- `interval` int — N периодов (≥ 1)
- `anchor_day` int (nullable) — день месяца для month/year; день недели
  (0–6) для week; null для day
- `start_date` date — первое возможное срабатывание
- `next_run_at` date — дата ближайшего планового срабатывания
- `mode` str(10) — autopost / remind
- `is_active` bool — дефолт true
- `end_date` date (nullable) — после этой даты правило не срабатывает
- `note` str(1000) (nullable)
- `created_by` uuid FK → users
- `created_at` timestamptz

### recurring_occurrences

- `id` uuid PK
- `workspace_id` uuid FK → workspaces
- `rule_id` uuid FK → recurring_rules
- `due_date` date — плановая дата сработавшего слота
- `amount` NUMERIC(20,4) — сумма на момент срабатывания (копия из правила)
- `status` str(10) — posted / pending / confirmed / skipped
- `transaction_id` uuid FK → transactions (nullable) — для posted/confirmed
- `created_at` timestamptz

Уникальный индекс `(rule_id, due_date)` — идемпотентность: повторный тик beat
не создаёт второе срабатывание того же слота. Индекс
`(workspace_id, status)` для выборки списка «к оплате».

### Семантика статусов

- `posted` — `autopost`: транзакция создана автоматически, `transaction_id`
  заполнен.
- `pending` — `remind`: ждёт решения пользователя, в списке «к оплате».
- `confirmed` — пользователь подтвердил `pending`: транзакция создана,
  `transaction_id` заполнен.
- `skipped` — пользователь пропустил `pending`: транзакция не создаётся.

## 4. Расписание и обработка (beat)

### Вычисление next_run_at

- От `start_date` шагами `interval` периодов; `next_run_at` — ближайший слот
  **строго в будущем** относительно текущей даты после обработки.
- Якорь: month/year — `anchor_day` число месяца; если в месяце меньше дней
  (например 31 в феврале) — кламп на последний день месяца. week —
  `anchor_day` как день недели. day — без якоря.
- Функция вычисления — чистая (без БД, детерминированная), покрыта юнит-
  тестами на краевые случаи.

### Задача scan_due_recurring (запускается beat)

Периодичность beat — каждые 15 минут (гранулярность дневная, частый тик лишь
уменьшает задержку). Задача по всем workspace:

1. Выбрать активные правила (`is_active`), где `next_run_at <= сегодня` и
   (`end_date` пуст или `next_run_at <= end_date`).
2. Для каждого правила создать `occurrence(due_date = next_run_at,
   amount = rule.amount)` (при конфликте по `(rule_id, due_date)` —
   пропустить, идемпотентность).
3. `autopost` → через `ledger.service` создать транзакцию (`source=recurring`,
   `account_id`/`category_id`/`amount`/`currency` из правила,
   `occurred_at = due_date`), записать `status=posted`, `transaction_id`.
   `remind` → `status=pending`.
4. Пересчитать `next_run_at` на ближайший будущий слот. **Пропущенные периоды
   не бэкфиллятся** — одно срабатывание на прогон, `next_run_at` перескакивает
   через все прошедшие слоты.

Задача идемпотентна и безопасна к повторному запуску. Работает межворкспейсно
(системный фон), но каждая создаваемая строка несёт `workspace_id` правила —
изоляция не нарушается.

## 5. Границы модулей и Celery

- Новый модуль `app/recurring/`: `models.py`, `schemas.py`, `repository.py`
  (весь SQL, фильтр `workspace_id`), `service.py` (доменная логика: расписание,
  обработка срабатываний, подтверждение/пропуск), `router.py`, `tasks.py`
  (тонкие Celery-задачи, зовущие service).
- Автопроведение и подтверждение создают транзакции **только через
  `ledger.service`**. Для этого `ledger.service` получает функцию постинга с
  параметром `source` (переиспользуется ручным вводом и регуляркой); прямых
  обращений `recurring` к `ledger.repository`/`ledger.models` нет.
- **import-linter**: `recurring.service`/`recurring.repository` не импортируют
  `identity` и `ledger.repository`/`ledger.models` (только `ledger.service`
  как публичный интерфейс); `identity` и `ledger` не импортируют `recurring`.
- **Celery**: `app/core/celery_app.py` — инстанс на Redis (брокер + result
  backend), расписание beat. Доменной логики в Celery нет — только вызов
  сервисных функций. Воркер и beat — отдельные процессы того же образа
  (команды в docker-compose); API-процесс отдельно.
- structlog: задачи beat пишут логи с `rule_id`/`workspace_id` (без PII/сумм).

## 6. API

Модуль `recurring`, роутер под `/api`. Все эндпоинты требуют
`require_workspace_member`; `workspace_id` — query-параметр.

### Правила

- `GET /api/recurring?workspace_id=` → список правил.
- `POST /api/recurring` `{account_id, category_id, amount, period, interval,
  anchor_day?, start_date, mode, end_date?, note?}` → 201. Валидация: знак
  `amount` vs `category.kind` (422); счёт и категория того же workspace (404);
  `period`/`mode` из допустимого набора; `interval ≥ 1`. `currency` со счёта;
  `next_run_at` вычисляется от `start_date`.
- `PATCH /api/recurring/{id}` `{amount?, interval?, anchor_day?, mode?,
  is_active?, end_date?, note?}` → 200. При изменении полей расписания
  `next_run_at` пересчитывается.
- `DELETE /api/recurring/{id}` → 204. Удаляет правило и его `occurrences`
  (FK `ON DELETE CASCADE`). Уже проведённые транзакции (`posted`/`confirmed`)
  — независимые записи ledger и **не удаляются**.

### Срабатывания

- `GET /api/recurring/occurrences?workspace_id=&status=pending` → список
  (по умолчанию pending; сортировка по `due_date`).
- `POST /api/recurring/occurrences/{id}/confirm` `{amount?}` → 201. Только для
  `pending`: создаёт транзакцию `source=recurring` (сумма — из тела, если
  передана, иначе из occurrence; знак сверяется с kind), `status=confirmed`,
  `transaction_id`. Повторный confirm/уже не pending → 409.
- `POST /api/recurring/occurrences/{id}/skip` → 200. Только для `pending`:
  `status=skipped`. Иначе 409.

## 7. Frontend

Поверх паттернов ledger (api-клиент, TanStack Query, Mantine, Zustand-стор
workspace).

- Пункт «Регулярные платежи» в навигации AppShell.
- **Правила**: таблица (счёт, категория, сумма, расписание в человекочитаемом
  виде «каждый месяц, 5-е», режим, активность) + модалка создания/правки
  (счёт, категория, сумма, период+interval+якорь, режим, start_date,
  end_date?, заметка). Тумблер `is_active`, удаление.
- **К оплате**: список `pending`/просроченных occurrences (правило, дата,
  сумма) с кнопками «Подтвердить» (в модалке — редактируемая сумма для ЖКХ)
  и «Пропустить».
- Суммы — строками с бэка; `Intl.NumberFormat`. Знак суммы правила выводится
  из kind категории, как в форме операции.

## 8. Тестирование

- Backend (pytest + testcontainers):
  - **чистая функция расписания**: next_run_at для day/week/month/year,
    interval > 1, кламп конца месяца (31 → последний день короткого месяца),
    старт в прошлом → ближайший будущий слот;
  - **beat scan**: autopost создаёт транзакцию (верный знак, `source=recurring`,
    `occurred_at=due_date`) и `next_run_at` перескакивает без бэкфилла;
    `remind` создаёт `pending`; `is_active=false`/просроченный `end_date` не
    срабатывают;
  - **идемпотентность**: повторный вызов scan не задваивает occurrence
    (unique `(rule_id, due_date)`);
  - confirm создаёт транзакцию (в т.ч. со скорректированной суммой), skip —
    нет; повторное действие → 409;
  - валидация правила: знак vs kind (422), чужой счёт/категория (404);
  - **изоляция workspace**: правила и occurrences чужого workspace недоступны
    (403/404);
  - Celery-задача тестируется прямым вызовом функции против тестовой БД (без
    запущенного воркера/брокера).
- Frontend (vitest): форма правила (валидация, знак), список «к оплате» с
  подтверждением/пропуском на замоканном API.
- import-linter прогоняется в CI; контракт для `recurring` не пустой
  (проверяется намеренной пробой при настройке).
- CI: worker/beat не запускаются в тестах; docker-джоба собирает образ с
  командами воркера/beat.

## 9. Принятые развилки

- Оба режима (`autopost`/`remind`); `remind` приземляется в in-app список
  «к оплате» (Telegram/почта — позже, к тому же механизму).
- Расписание «каждые N <период>» + якорь (не cron/RRULE) — покрывает домашние
  кейсы, простое в UI и валидации.
- Просрочка: провести один раз и перескочить `next_run_at` на ближайший
  будущий слот; бэкфилл пропущенных периодов не делаем (защита от пачки
  транзакций и дублей с ручным вводом).
- Материализованные `recurring_occurrences` — единый механизм для обоих
  режимов, аудит-история, идемпотентность через unique `(rule_id, due_date)`.
- Остановка правила: `is_active` + опциональный `end_date` (без счётчика
  срабатываний).
- Регулярка проводит транзакции только через публичный `ledger.service`
  (source-параметр), не через таблицы ledger.
- Celery на существующем Redis; воркер и beat — отдельные процессы того же
  образа.

## 10. Открытые вопросы

- Таймзона: `next_run_at`/`due_date` — даты, сравнение по дате сервера; явная
  таймзона workspace не вводится (одно домохозяйство). Пересмотреть при B2B.
- Виджет «предстоящие платежи» на дашборде — возможен позже; в B1 список живёт
  на странице «Регулярные платежи», чтобы не расширять дашборд.
- Реальная периодичность beat (15 мин) может быть уточнена при деплое.
