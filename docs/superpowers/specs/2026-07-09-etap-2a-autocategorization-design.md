# AIccountant — этап 2a «AI-автокатегоризация операций (v1: LLM-классификатор)»

Дата: 2026-07-09. Статус: на ревью.

Первый под-проект этапа 2 (AI-конвейер). Идёт после этапа 1 (ledger,
регулярка, импорт — смержено). Общая спека:
`docs/superpowers/specs/2026-07-02-aiccountant-design.md` (разделы 5.3–5.4).

## 1. Цель

Автоматически проставлять категорию операции по её описанию (мерчанту),
чтобы не делать это руками — в духе курса на автоматизацию. Категория
опциональна (как заметка), поэтому ошибка не критична, но важны доверие
и обучение на правках.

Мастер-спека уже зафиксировала подход этапа 2 (свой провайдеро-независимый
LLM-слой; категоризация с kNN по pgvector и фидбек-лупом; eval в CI). Эти
развилки не пересматриваются. Настоящий под-проект — **стадирование**: v1 —
чистый LLM-классификатор с лёгким фидбек-лупом (few-shot из подтверждённых
правок) без pgvector. kNN по pgvector и реэмбеддинг — следующий под-проект.

## 2. Скоуп

Включено:

- Новый модуль `app/ai/`: интерфейс `LLMClient` (structured output) +
  OpenAI-совместимая реализация (настраиваемый `base_url`), фабрика по конфигу.
- Доменная логика категоризации в `app/ledger/` (пишет `transactions`,
  читает `categories` — таблицы ledger): `ledger/categorization.py` +
  Celery-задача `ledger/tasks.py`.
- Миграция `0006`: поля `category_confirmed`, `category_confidence`,
  `suggested_category_id` в `transactions`.
- Гибридная простановка: авто выше порога уверенности, подсказка-на-
  подтверждение ниже порога.
- Автотриггер задачи после импорта и создания ручной операции без категории;
  ручной перезапуск по кнопке (`POST /api/transactions/categorize`).
- Фидбек-луп v1: few-shot из последних подтверждённых пар `merchant→категория`
  этого workspace.
- Frontend: бейдж «AI» у авто-категории, чип-подсказка с подтверждением/
  сбросом, кнопка «Категоризировать без категории».
- Eval-набор с метрикой accuracy; детерминированные тесты логики в CI.
- import-linter: контракт `ledger → ai` (и `ai` ни от кого не зависит).

Явно исключено (следующие под-проекты этапа 2 или позже):

- pgvector, эмбеддинги, kNN, реэмбеддинг-задача (`EmbeddingClient`,
  `Transcriber` в модуле `ai` **не строятся** — YAGNI).
- LLM-парсер произвольных выписок; Telegram-бот; чат-агент; аномалии.
- Батчинг нескольких операций в один LLM-вызов (возможная оптимизация позже).
- Создание новых категорий классификатором — выбор только из существующих.

## 3. Модули и границы

- **`app/ai/`** — только LLM-инфраструктура, без доменного знания:
  - `LLMClient` (Protocol): метод structured-output (JSON по схеме);
  - `OpenAICompatLLMClient` — реализация через OpenAI-совместимый эндпоинт
    (`LLM_BASE_URL`, `LLM_API_KEY`, модель per-вызов). Вендорский SDK/HTTP —
    только здесь.
  - Фабрика `build_llm_client()` по конфигу. Дефолт — облачный провайдер для
    демо; локальный Ollama работает тем же OpenAI-совместимым интерфейсом
    (иной `base_url`) и документируется как приватная альтернатива.
- **`app/ledger/`** — категоризация как доменная логика ledger:
  - `categorization.py` — сборка кандидатов, промпт, порог, применение;
  - `tasks.py` — Celery-задача `categorize_workspace`.
  - Вызывает `ai.LLMClient` через интерфейс (`ledger → ai` — разрешённое
    направление; `ai` не импортирует ledger). Простановка категории — через
    существующий `ledger.service` (инвариант знак↔kind уже там).

## 4. Данные (миграция `0006`)

`transactions` получают:

- `category_confirmed BOOLEAN NOT NULL DEFAULT false` — категорию подтвердил
  человек (ручная простановка/подтверждение подсказки = `true`; авто-
  простановка AI = `false`);
- `category_confidence NUMERIC(4,3) NULL` — уверенность последнего прогона
  (0..1). `NUMERIC`, не `float` — по стилю проекта (float для чисел из БД
  не заводим даже вне денег);
- `suggested_category_id UUID NULL` FK→`categories` — предложение ниже порога
  (категория ещё не применена).

Состояния строки:

| Состояние | `category_id` | `suggested_category_id` | `category_confirmed` |
|---|---|---|---|
| без категории | null | null | false |
| AI-подсказка (ниже порога) | null | задан | false |
| AI авто (выше порога) | задан | null | false |
| человек (ручная/подтверждён.) | задан | null | true |
| подсказка отклонена | null | null | true |

Переходы: подтверждение подсказки → `category_id := suggested`,
`confirmed := true`, `suggested := null`. Переопределение (ручной выбор любой
категории) → `category_id := выбор`, `confirmed := true`, `suggested := null`.
Отклонение подсказки (dismiss) → `suggested := null`, `confirmed := true`
(категория остаётся пустой): отклонение — это решение человека, поэтому
операция помечается принятой и **не попадает в повторную категоризацию**
(иначе классификатор предлагал бы то же снова).

## 5. Классификатор (поток)

1. Задача выбирает операции workspace без категории, без активной подсказки
   и без принятого человеком решения
   (`category_id IS NULL AND suggested_category_id IS NULL AND category_confirmed IS FALSE`).
   Последнее условие исключает отклонённые подсказки из повторных прогонов.
2. Кандидаты для операции = категории workspace нужного `kind` по знаку суммы
   (`amount < 0` → `expense`, иначе `income`). Плюс few-shot: последние K
   (напр. 10) операций workspace с `category_confirmed = true` и непустым
   `merchant` — пары `merchant → имя категории`. Это фидбек-луп v1: правки
   пользователя попадают в промпт и улучшают последующую точность.
3. `ai.LLMClient` вызывается со structured-output-схемой и возвращает
   `{category_name: str | null, confidence: Decimal}` (0..1). Промпт содержит
   описание операции, знак/kind и список допустимых имён категорий.
4. Валидация кодом: `category_name` должно быть среди кандидатов; иначе —
   трактуем как «нет категории» (никаких новых категорий).
5. Порог `CATEGORIZE_CONFIDENCE_THRESHOLD` (дефолт `0.8`):
   - `confidence ≥ порог` и кандидат валиден → авто-простановка
     (`category_id`, `confirmed = false`, `category_confidence`);
   - `confidence < порог`, но кандидат валиден → `suggested_category_id`,
     `category_confidence`;
   - кандидата нет → операция остаётся без категории.
6. Один LLM-вызов на операцию (KISS). Латентность не критична — фон.

## 6. Запуск и API

- Celery-задача `categorize_workspace(workspace_id)` — воркер/beat уже подняты.
- Автотриггер (постановка задачи в очередь через `ledger.service`):
  - после `imports.commit_import` (imports дёргает только `ledger.service` —
    границы целы);
  - после создания ручной операции без категории.
- `POST /api/transactions/categorize?workspace_id=` — ручной перезапуск
  (та же задача), ответ `202`. Требует `require_workspace_member`.
- Подтверждение/переопределение подсказки — через существующий
  `PATCH /api/transactions/{id}`: установка `category_id` ставит
  `confirmed = true` и чистит `suggested_category_id`.

## 7. Конфигурация

Настройки в `app/core/config`:

- `LLM_BASE_URL`, `LLM_API_KEY` — эндпоинт (облако по умолчанию; Ollama —
  иной `base_url`);
- `LLM_MODEL_CATEGORIZE` — модель для категоризации (дешёвая);
- `CATEGORIZE_CONFIDENCE_THRESHOLD` — порог (дефолт `0.8`);
- `CATEGORIZE_FEWSHOT_LIMIT` — сколько подтверждённых примеров в промпт
  (дефолт `10`).

Ключ — в `.env` (в `.gitignore`), в репозиторий не попадает.

## 8. Frontend

- Лента операций: у авто-категории (`confirmed = false`, `category_id` задан)
  — бейдж «AI»; у подсказки (`suggested_category_id`) — чип «AI: <категория>»
  с кнопками подтвердить (✓) / сбросить (✗); ручной выбор категории —
  переопределение.
- Подтверждение/сброс/выбор идут через `PATCH /transactions/{id}`.
- Кнопка «Категоризировать без категории» → `POST …/categorize`, мягкий
  статус «разбираем…», по завершении — обновление ленты (перезапрос).
- Бакет «Без категории» на дашборде уже есть (этап опциональной категории).

## 9. Ошибки

- LLM недоступен, таймаут или невалидный/непарсящийся ответ → операция просто
  остаётся без категории; задача логирует событие (только идентификаторы, без
  PII и сумм), не падает и не глотает исключение молча.
- Частичный сбой в пачке не откатывает уже проставленные — каждая операция
  обрабатывается и коммитится независимо (или маленькими партиями).

## 10. Тестирование

- Backend (pytest):
  - **логика классификации** на **записанных** ответах LLM (детерминированно,
    в CI, без ключа): порог → авто/подсказка/пусто; валидация имени кандидата;
    фильтр кандидатов по знаку↔kind; сборка few-shot;
  - **контрактный тест `LLMClient`** на записанном ответе (форма structured
    output, разбор в модель);
  - **изоляция workspace**: категоризация не читает/не пишет чужие операции
    и категории;
  - **автотриггер**: после `commit_import` и ручного создания без категории
    задача поставлена в очередь (мок брокера);
  - **переходы состояний** через `PATCH` (подтверждение/переопределение чистит
    suggestion, ставит `confirmed`).
- **Eval-набор** (требование CLAUDE.md): фикстуры `(описание, знак, ожидаемая
  категория)` + раннер, считающий accuracy. В CI прогоняется детерминированная
  логика на записях; прогон accuracy против **живого** провайдера — отдельный
  opt-in job, пропускается без ключа (в CI без ключа — skip, не падение).
- Frontend (vitest): рендер бейджа «AI», чипа-подсказки и кнопок; подтверждение
  шлёт `PATCH`; кнопка категоризации шлёт `POST …/categorize`.
- import-linter: контракт — `ledger` может импортировать `ai` (интерфейс);
  `ai` не импортирует доменные модули; проверяется пробоем.

## 11. Принятые развилки

- Стадирование: v1 — чистый LLM-классификатор + few-shot-фидбек; pgvector-kNN
  и реэмбеддинг — следующий под-проект (подход мастер-спеки сохранён).
- Провайдеро-независимый слой; дефолт — облачный OpenAI-совместимый для демо,
  Ollama — приватная альтернатива тем же интерфейсом (конфиг).
- Гибридная UX: авто выше порога (помечено «AI», обратимо), подсказка ниже.
- Классификация — Celery фоном; автотриггер после импорта/создания + ручная
  кнопка.
- Кандидаты — только существующие категории workspace нужного `kind`; новые
  категории классификатор не создаёт.
- Категоризация живёт в `ledger` (её таблицы), использует `ai.LLMClient` через
  интерфейс; модуль `ai` доменно-независим.
- `EmbeddingClient`/`Transcriber` в v1 не строятся (YAGNI).

## 12. Открытые вопросы

- Self-reported confidence чистого LLM шумная — порог консервативный, много
  пойдёт в подсказки. Настоящая калибр. уверенность (kNN-доля) — на pgvector-
  этапе; тогда же пересмотрим порог.
- Значение порога и `few-shot`-лимита уточняются по eval-набору на реальных
  описаниях Т-Банка.
- Батчинг операций в один вызов — если латентность/стоимость станут заметны.
- Операции без `merchant` сейчас всё равно проходят через LLM (пустое описание →
  почти всегда «нет категории»): тратится лишний вызов. Если станет заметно —
  фильтровать `merchant IS NOT NULL` в выборке некатегоризированных.
- `category_confidence` пишется и при решении «нет категории» (последняя
  попытка) — на состояние строки не влияет, оставлено как след прогона.
