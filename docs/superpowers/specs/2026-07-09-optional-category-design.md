# AIccountant — «Опциональная категория» (операции + правила)

Дата: 2026-07-09. Статус: на ревью.

Правка поведения существующих модулей `ledger` (этап 1.2a) и `recurring`
(этап 1.2b1). Мотив: продукт ориентирован на автоматизацию — обязательная
категория на каждую операцию противоречит принципу «не делать руками то, что
можно не делать». Категория становится опциональным тегом (как заметка);
некатегоризированное позже разгребёт AI-категоризация (этап 2).

## 1. Цель

Сделать категорию **необязательной** при создании операций (транзакций) и
правил регулярных платежей. Направление (расход/доход) перестаёт выводиться из
`category.kind` и становится явным выбором пользователя; знак суммы (уже
знаковой) кодирует направление, отдельная колонка не вводится.

## 2. Скоуп

Включено:

- Транзакции: `category_id` опционален при создании/правке; направление задаёт
  пользователь (знак суммы).
- Правила регулярки: `category_id` опционален; автопроведение создаёт
  транзакцию с той же (возможно пустой) категорией.
- Дашборд: некатегоризированные расходы месяца сводятся в бакет «Без категории».
- Frontend: переключатель Расход/Доход в формах операции и правила; поле
  «Категория» опционально и фильтруется по направлению.
- Миграция `0004`: `recurring_rules.category_id` → nullable.

Явно исключено:

- Очистка уже проставленной категории через PATCH (сейчас `null` в теле PATCH =
  «не менять»; отдельный сентинел — позже, если понадобится).
- Изменение семантики `category.kind`, дерева категорий, seed-набора.
- AI-категоризация (этап 2) — здесь только задел (пустая категория).

## 3. Модель данных

- `transactions.category_id` — **уже nullable** (у переводов пусто); изменений
  в схеме нет. Некатегоризированная обычная операция: `category_id = null` И
  `transfer_group_id = null` (отличима от перевода).
- `recurring_rules.category_id` — сейчас NOT NULL. Миграция `0004_recurring_optional_category`
  через `alembic revision --autogenerate` снимает NOT NULL (FK на `categories.id`
  сохраняется).

## 4. Backend — валидация проведения

Единая точка `ledger.service.validate_posting` (её зовут ручной ввод,
автопостинг регулярки, подтверждение срабатывания и валидация правила).
Параметр `category_id` становится `uuid.UUID | None`. Логика:

- счёт обязателен и принадлежит workspace, иначе `NotFoundError` (404);
- `amount != 0`, иначе `SignMismatchError` (422);
- **если `category_id` задан** — категория существует в этом workspace (иначе
  `NotFoundError`) и её `kind` соответствует знаку суммы (расход `< 0`,
  доход `> 0`), иначе `SignMismatchError`;
- если `category_id` пуст — проверки категории пропускаются, `currency` берётся
  со счёта как обычно.

Затрагивает: `TransactionCreate.category_id` и `RuleCreate.category_id` →
`uuid.UUID | None = None`; `post_transaction`/`create_transaction` пробрасывают
опциональную категорию; `update_transaction` перестаёт требовать категорию
(текущая проверка «обычная операция всегда имеет категорию» снимается — `null`
валиден; знак сверяется с kind только когда итоговая категория не пуста);
`RuleOut.category_id`/`TransactionOut.category_id` — уже опциональны.

Инвариант знака при заданной категории сохраняется (тест на 422 остаётся).

## 5. Backend — дашборд

`repository.month_expenses_by_category` переходит с INNER на **LEFT JOIN**
категории и группируется по `Transaction.category_id`:

- условия прежние: `amount < 0`, `transfer_group_id IS NULL`, `occurred_at` в
  текущем месяце (полуинтервал);
- группировка по `Transaction.category_id` (и имени категории через outerjoin);
- строка с `category_id = null` → `category_name = "Без категории"`.

`MonthExpense.category_id` → `uuid.UUID | None`. Сумма бакета «Без категории»
считается как у остальных (`SUM(-amount)`), сортировка по убыванию сохраняется.
Переводы по-прежнему исключены.

## 6. Frontend

`TransactionForm` и `RecurringRuleForm`:

- Добавляется **сегмент-контрол «Расход / Доход»** (по умолчанию «Расход») —
  задаёт знак суммы вместо `category.kind`. Пользователь вводит величину суммы;
  знак применяется из переключателя.
- Поле «Категория» — **опциональное** (`clearable`); список категорий
  фильтруется по выбранному направлению (`kind === 'expense'` для «Расход»,
  `'income'` для «Доход») — нельзя выбрать несоответствующую категорию.
- `merchant`/`note`/`category_id` — необязательные; пустые не отправляются.

Дашборд рендерит бакет «Без категории» как обычную строку расходов (данные с
бэка, суммы строками, `Intl.NumberFormat`). Отдельного кода для него не нужно.

## 7. Тестирование

- Backend (pytest):
  - создание операции **без категории** → 201, `category_id = null`;
  - расход без категории **попадает в бакет «Без категории»** на дашборде;
  - страховка знака при заданной категории (расход с доходной категорией → 422)
    сохраняется; `amount == 0` → 422;
  - создание правила регулярки **без категории** → 201; автопроведение создаёт
    некатегоризированную транзакцию (`source=recurring`, `category_id = null`);
  - подтверждение remind-срабатывания правила без категории создаёт
    некатегоризированную транзакцию;
  - миграция `0004` применяется, `recurring_rules.category_id` — nullable
    (`alembic check` подтверждает совпадение с моделью).
- Frontend (vitest):
  - форма операции отправляет **отрицательный** amount при «Расход» без
    выбранной категории; **положительный** при «Доход»;
  - список категорий в форме фильтруется выбранным направлением.

## 8. Принятые развилки

- Направление кодируется знаком суммы (без отдельной колонки direction) —
  минимальная модель, миграции под направление не нужно.
- Категория — опциональный тег во всех точках ввода (операции и правила),
  единообразно.
- При заданной категории её `kind` обязан соответствовать направлению; в UI
  это гарантируется фильтрацией списка, на бэке — страховкой (422).
- Некатегоризированные расходы видны на дашборде в бакете «Без категории», не
  теряются в отчёте; это же — задел под AI-категоризацию.

## 9. Открытые вопросы

- Очистка категории у существующей операции/правила через PATCH — отложена
  (нужен сентинел, чтобы отличить «не менять» от «сбросить в null»).
- Модель `recurring_rules` могла бы позже получить явную валидацию, что при
  autopost без категории направление всё равно однозначно (обеспечено знаком
  `amount`) — сейчас это уже так.
