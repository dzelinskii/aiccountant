# moneyrain — дизайн-документ

Дата: 2026-07-02. Статус: на ревью (v2 — фокус на AI-анализ, мультивалютная
конвертация отложена).

## 1. Видение

AI-бухгалтер для домохозяйства — не ещё один трекер доходов/расходов.
Ядро продукта — анализ и автоматизация:

- сам читает финансовые документы: выписки банков, чеки, квитанции;
- сверяет документы с транзакциями и находит расхождения (чек не совпал
  со списанием, дубль списания, чек без транзакции);
- находит аномалии: всплеск трат по категории, подорожавшая подписка,
  необычный мерчант;
- прогнозирует расходы и отвечает «хватит ли до конца месяца»;
- напоминает о регулярных платежах и неоплаченных квитанциях;
- помогает распределять бюджет по категориям на основе истории;
- отвечает на вопросы о деньгах в чате и в Telegram (текст/голос/фото чека).

Учёт (счета, категории, транзакции) — сознательно минимальный фундамент
данных, на котором стоит анализ.

Двойная цель проекта: рабочий продукт для себя и близких + портфолио,
осознанно демонстрирующее backend-инструменты (Celery, Redis, pgvector,
Kafka, наблюдаемость) и AI-инженерию (LLM-конвейеры, агент с инструментами,
evals) там, где у них есть честная задача.

## 2. Пользователи и режим работы

- Пользователи на старте: владелец + близкие. Multi-user с ролями с первого
  дня, но без публичного онбординга, биллинга и тарифов.
- Развёртывание: self-host через docker-compose + живое демо на VPS
  (демо-логин и сид-данные для рекрутёров).
- B2B-шов: доменная модель допускает поздний виток в B2B (ИП/самозанятые)
  без переписывания ядра — `workspace.type` и расширяемые роли. Сам B2B
  вне скоупа до этапа 5.
- Коммерциализация — «дверь открыта»: не драйвер решений, но архитектура
  путь к платному SaaS не закрывает.

## 3. MVP-скоуп (этап 1)

Минимальный фундамент данных, чтобы этапами 2–4 строить анализ и AI.

Включено:

- Auth: email + пароль (argon2), httpOnly cookie-сессии + CSRF.
- Workspace «домохозяйство», участники, роли owner/member.
- Счета: карта/наличные/накопительный. Поле `currency` есть (дефолт RUB),
  но никакой конвертации — дашборд показывает суммы раздельно по валютам.
- Категории: дерево, дефолтный набор при создании workspace.
- Транзакции: расход/доход/перевод, быстрый ручной ввод.
- Регулярные платежи: правило с расписанием → автопроведение или напоминание.
- Импорт выписки Т-Банка (CSV, детерминированный парсер) с дедупликацией.
- Дашборд: остатки, расходы месяца по категориям, лента операций.
- PWA (installable), деплой на VPS с TLS, CI, структурные логи, демо-режим.

Явно исключено из MVP:

- мультивалютная конвертация, курсы валют, сводная аналитика между валютами
  (поздняя опция — модель готова, machinery отложена);
- AI-фичи, Telegram-бот, Kafka, Prometheus/Grafana — появляются этапами;
- бюджеты, прогнозы, сверка — этапы 3–4;
- B2B, нативные клиенты.

## 5. Архитектура

### 5.1 Модульный монолит

Один репозиторий (`backend/`, `frontend/`, `infra/`), одна кодовая база Python.
Модули: `identity`, `ledger`, `recurring`, `imports`, `ai`, `analytics`
(сверка, аномалии, прогноз, бюджеты), `notifications`. Правила границ:

- модули общаются через сервисные интерфейсы, в чужие таблицы не ходят;
- границы проверяет import-linter в CI;
- API-процесс и Celery-воркеры — разные процессы одного Docker-образа.

### 5.2 Модель данных

Ядро (все доменные таблицы несут `workspace_id` — row-level tenancy):

- `users(id, email, password_hash, created_at)`
- `workspaces(id, name, type)` — `type='personal'`, B2B-шов
- `memberships(user_id, workspace_id, role)`
- `accounts(id, workspace_id, name, type, currency, is_archived)`
- `categories(id, workspace_id, parent_id, name, kind)` — kind: income/expense
- `transactions(id, workspace_id, account_id, category_id, amount, currency,
  occurred_at, merchant, note, source, external_id, import_id,
  transfer_group_id, created_by)` — source: manual/import/telegram/recurring
- `recurring_rules(id, workspace_id, account_id, category_id, amount,
  schedule, mode, next_run_at, is_active)` — mode: autopost/remind
- `imports(id, workspace_id, file_name, bank_profile, status, stats, created_by)`

Появляются позже:

- `merchant_embeddings(text, embedding, model, category_id, workspace_id)` —
  этап 2;
- `documents(id, workspace_id, kind, source, total, occurred_at, merchant,
  items, matched_transaction_id, discrepancy, status)` — чеки и квитанции,
  этап 3; kind: receipt/utility_bill, source: qr/vision/upload;
- `budgets(id, workspace_id, category_id, period, limit_amount)` — этап 4;
- `events_outbox` — этап 4.

Доменные тонкости:

- Деньги — `NUMERIC(20, 4)`, никаких float.
- `currency` на счёте и транзакции хранится с первого дня (дефолт RUB) —
  это страховка от болезненной миграции, если мультивалютность вернётся;
  конвертации и курсов в ядре нет.
- Перевод — пара транзакций с общим `transfer_group_id`.
- Дедупликация импорта: `external_id` от банка, иначе хеш
  (дата + сумма + описание); уникальный индекс на `(account_id, external_id)`.
- Индексы: `(workspace_id, occurred_at)`, `(account_id, occurred_at)`.

### 5.3 LLM-слой: независимость от провайдера

Принцип: никакой привязки к конкретному вендору. В модуле `ai` — свои
интерфейсы (Protocol):

- `LLMClient` — чат, tool use, structured output;
- `EmbeddingClient` — эмбеддинги;
- `Transcriber` — речь → текст.

Дефолтная реализация `LLMClient`/`EmbeddingClient` — OpenAI-совместимый
формат с конфигурируемым `base_url`: работает с OpenAI, OpenRouter, DeepSeek,
Gemini (compat-эндпоинт), локальными Ollama/vLLM; Anthropic — через OpenRouter.
Модель задаётся конфигом per-фича (дешёвая — категоризация, сильная — агент):
`LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL_CATEGORIZE`, `LLM_MODEL_AGENT`,
`EMBED_MODEL` и т.п.

Эмбеддинги: размерность фиксируется при настройке; имя модели хранится
в каждой строке; смена модели = Celery-задача реэмбеддинга.

`Transcriber`: дефолт — faster-whisper (small) на CPU (бесплатно, приватно,
без вендора); облачный STT — альтернативная реализация.

### 5.4 AI-фичи

- Парсинг документов (этап 2): LLM получает текст из PDF/CSV неизвестного
  формата (выписки, позже квитанции) → строго типизированный JSON; валидация
  сумм и итогов кодом; невалидное — в ручной разбор, не молча.
- Категоризация (этап 2): сначала kNN по pgvector среди прошлых
  подтверждённых решений workspace; LLM — только при низкой уверенности;
  ручные исправления пополняют выборку (фидбек-луп).
- Чат-агент (этап 3): свой цикл на tool use. Инструменты: `sql_query`
  (read-only роль Postgres, allowlist таблиц, обязательный LIMIT, timeout),
  `spending_stats`, `semantic_search`, `upcoming_payments`, `find_anomalies`.
  Один агент в вебе и Telegram.
- Объяснение аномалий и расхождений (этапы 3–4): цифры считает
  детерминированный код, LLM формулирует человекочитаемое «что случилось
  и почему это странно».
- Eval-набор: размеченные транзакции + метрика точности категоризации,
  прогоняется в CI.

### 5.5 Чеки и голос (этап 3)

Чеки: фото → сначала локальное декодирование фискального QR (pyzbar) —
у российских чеков сумма/дата достаются детерминированно и бесплатно;
позиции чека — опционально через API ФНС (неофициальный, помечаем как
эксперимент); фолбэк для нечитаемых чеков — vision-модель через тот же
`LLMClient`. Голос: Telegram voice → `Transcriber` → тот же текстовый
парсер («кофе 450 рублей»).

### 5.6 Анализ и проактивность (ядро продукта, этапы 3–4)

- Сверка (этап 3): документ (чек/квитанция) матчится с транзакцией по
  сумме/дате/мерчанту с допусками; расхождения — отдельная лента «на разбор»:
  сумма чека ≠ списанию, дубль списания, чек без транзакции, транзакция
  без чека по мерчантам, где чеки обычно есть.
- Аномалии (этап 3): детерминированная статистика — всплеск по категории
  относительно скользящего среднего, изменение суммы регулярного платежа
  или подписки, новый крупный мерчант; LLM только объясняет.
- Прогноз расходов (этап 4): регулярка + скользящие средние по категориям;
  «хватит ли до конца месяца», горизонты 30/90 дней. Детерминированная
  математика, без ML-мистики.
- Распределение бюджета (этап 4): лимиты по категориям; AI предлагает лимиты
  из истории; уведомления при приближении к лимиту.
- Напоминания (этапы 1→4): регулярные платежи с этапа 1; с этапа 4 — умные
  (неоплаченная квитанция, аномалия, прогнозируемый кассовый разрыв)
  через событийный слой.

### 5.7 Событийный слой (этап 4)

- Transactional outbox: доменные события пишутся в `events_outbox` в той же
  транзакции, что и данные; publisher переносит их в Kafka.
- Топик `transaction-events`; консьюмеры: notifications (проактивные
  Telegram-алерты), projections (агрегаты для дашборда и прогноза),
  audit (журнал).
- Консьюмеры идемпотентны; replay-демо: перестроить проекции с нуля одной
  командой.
- Деплой — Redpanda (Kafka-совместимый, один бинарь); в README честно:
  на этих объёмах хватило бы Redis Streams, Kafka введена осознанно,
  но консьюмеры у неё настоящие.

### 5.8 Безопасность

- argon2, httpOnly cookie-сессии, CSRF, rate limiting на Redis.
- Изоляция по `workspace_id` в repository-слое (MVP); Postgres RLS как
  защита в глубину — этап 3 (hardening).
- TLS через Caddy; секреты вне git; ежедневный pg_dump в S3-совместимое
  хранилище.
- Telegram: линковка аккаунта одноразовым кодом, secret token вебхука,
  идемпотентность по update_id.
- Логи без PII и сумм — только идентификаторы.
- Аудит входов и критичных действий.

## 6. Дорожная карта

Дисциплина вместо дедлайна: этап закрыт, только когда задеплоен на демо-URL
и описан в README. Следующий не начинается раньше.

### Этап 0 — фундамент
Репозиторий, скелет FastAPI + React + docker-compose, CI (ruff, mypy, pytest,
build), деплой пустого приложения на VPS с TLS.
Веха: «hello» живёт по URL, пайплайн зелёный.

### Этап 1 — минимальное ядро учёта
Скоуп раздела 3. Демонстрирует: FastAPI/SQLAlchemy 2/Alembic, честные
Celery + Redis, PWA, CI/CD.
Веха: владелец и близкие реально ведут финансы в приложении.

### Этап 2 — AI-конвейер документов
LLM-парсер PDF/произвольных CSV (выписки Сбера и др.), автокатегоризация
с pgvector и фидбек-лупом, Telegram-бот (линковка, текстовый ввод,
уведомления), провайдеро-независимый LLM-слой, eval-набор категоризации.
Демонстрирует: structured output, эмбеддинги, pgvector, aiogram, evals.
Веха: PDF-выписка Сбера → категоризированные транзакции за минуту;
трата вносится из Telegram за пять секунд.

### Этап 3 — ассистент, сверка и аномалии
Чат-агент с инструментами (web + Telegram), голосовой ввод (faster-whisper),
чеки и квитанции (QR-first + vision-фолбэк), сверка документов
с транзакциями и лента расхождений, детект аномалий, семантический поиск,
Prometheus + Grafana, дашборд стоимости LLM, Sentry, RLS-hardening.
Демонстрирует: агентную оркестрацию своими руками, text-to-SQL с guardrails,
мультимодальность (vision, STT), reconciliation-алгоритмы, observability.
Веха: сфотографировал чек — система нашла списание и подсветила
расхождение; «сколько мы потратили на еду в мае против апреля?» — ответ
с цифрами; Grafana доступна в демо.

### Этап 4 — проактивность и событийный слой
Прогноз расходов и кэшфлоу, бюджеты с AI-предложением лимитов, умные
напоминания. Outbox → Kafka (Redpanda) → консьюмеры
notifications/projections/audit, replay-демо.
Демонстрирует: event-driven дизайн, идемпотентные консьюмеры, replay,
прогнозную аналитику.
Веха: приложение само предупреждает «к 25-му числу может не хватить
на аренду»; уведомления и проекции работают через шину; README
объясняет «зачем».

### Этап 5 — опции (каждая — отдельное решение)
Мультивалютная конвертация (курсы, сводная аналитика между валютами);
публичный онбординг и биллинг; B2B-виток (workspace type=business, роль
«бухгалтер», экспорт отчётов, сущности для ИП); n8n-showcase (почта →
импорт); React Native.

## 8. Не-функциональные требования

- Тесты: pytest + pytest-asyncio; testcontainers (Postgres/Redis) для
  интеграционных; юниты на доменную математику (расписания, дедуп, сверка,
  аномалии); LLM-слой — контрактные тесты на записанных ответах +
  eval-набор; фронт — vitest, Playwright на 2–3 смоук-сценария.
- CI/CD: GitHub Actions — lint, типы, тесты, сборка образов, деплой по тегу.
- Наблюдаемость: structlog (JSON, request_id, workspace_id) с этапа 1;
  Prometheus/Grafana/Sentry — этап 3.
- README: диаграмма архитектуры (mermaid), скриншоты, демо-URL и демо-логин,
  раздел «решения и почему» в ADR-стиле, включая честные пометки про Kafka.

## 9. Принятые развилки

- Фокус продукта: анализ и AI (сверка, аномалии, прогноз, распределение);
  учёт — минимальный фундамент.
- Мультивалютная конвертация: отложена до этапа 5; поля `currency`
  в модели есть с первого дня, конвертация добавится без переписывания.
- UI-кит: Mantine (data-heavy UI собирается быстрее).
- Auth: cookie-сессии, не JWT (один бэкенд, PWA).
- Python-тулинг: uv + ruff + mypy.
- Хостинг: любой VPS с Docker; архитектура от выбора не зависит.
- RLS: не в MVP, этап 3.
- LLM: OpenAI-совместимый формат за своим Protocol; без LangGraph и LiteLLM.
- Без Nx, Qdrant, RabbitMQ; n8n — только как поздняя опция вне ядра.
