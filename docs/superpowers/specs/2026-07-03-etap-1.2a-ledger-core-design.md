# AIccountant — этап 1.2a «Ядро учёта (ledger core)»

Дата: 2026-07-03. Статус: на ревью.

Под-проект этапа 1 (ledger), часть A. Общая спека продукта:
`docs/superpowers/specs/2026-07-02-aiccountant-design.md` (раздел 3 —
MVP-скоуп). Часть B (Celery-воркер, регулярные платежи, импорт CSV) —
отдельная спека, начинается после мержа A.

## 1. Цель

Полноценный ручной трекер финансов домохозяйства: счета в нескольких
валютах (без конвертации), категории-дерево, транзакции (расход/доход/
перевод), базовый дашборд. После этой части владелец и близкие уже реально
ведут финансы в приложении вручную; автоматизация (регулярка, импорт
выписок) добавляется частью B.

Строится поверх identity (этап 1.1): auth, workspace, роли, membership.

## 2. Скоуп

Включено:

- Счета: создание, переименование, архивация; тип (card/cash/savings),
  валюта (дефолт RUB, без конвертации).
- Категории: дерево (parent_id), kind income/expense, дефолтный набор при
  создании workspace, создание/правка.
- Транзакции: расход и доход (одна запись), перевод между своими счетами
  (пара записей), ручной ввод, правка, удаление, лента с фильтрами и
  пагинацией.
- Дашборд: остатки по счетам, расходы текущего месяца по категориям, лента
  последних операций — один агрегирующий эндпоинт.
- Frontend: AppShell с навигацией, страницы Дашборд/Счета/Категории/
  Операции на Mantine.
- Изоляция по workspace_id в repository-слое; import-linter для границ
  модулей; structlog с workspace_id в контексте запроса.

Явно исключено (часть B или позже):

- Celery-воркер, регулярные платежи, импорт CSV/выписок, дедупликация,
  колонки external_id/import_id в transactions.
- Мультивалютная конвертация, курсы, сводная аналитика между валютами
  (этап 5; поле currency в модели есть, machinery нет).
- Графики, тренды, сравнение периодов (аналитика этапов 3–4).
- AI-фичи.

## 3. Модель данных

Все таблицы несут `workspace_id`. Деньги — `NUMERIC(20, 4)`, `Decimal`
в коде, `float` запрещён. Миграция `0002_ledger` — через
`alembic revision --autogenerate` с ручной вычиткой имён/порядка.

### accounts

- `id` uuid PK
- `workspace_id` uuid FK → workspaces
- `name` str(200)
- `type` str(20) — card / cash / savings
- `currency` str(3) — дефолт "RUB"
- `is_archived` bool — дефолт false
- `created_at` timestamptz

### categories

- `id` uuid PK
- `workspace_id` uuid FK → workspaces
- `parent_id` uuid FK → categories (nullable — дерево)
- `name` str(200)
- `kind` str(20) — income / expense
- `created_at` timestamptz

Дефолтный набор создаётся при регистрации (см. раздел 5 — вызывается из
роутера регистрации, не из сервиса identity). Набор: расходные — «Еда»,
«Транспорт», «Жильё», «Связь», «Развлечения», «Здоровье», «Прочее»; доходные
— «Зарплата», «Прочие доходы».

### transactions

- `id` uuid PK
- `workspace_id` uuid FK → workspaces
- `account_id` uuid FK → accounts
- `category_id` uuid FK → categories (nullable — у переводов null)
- `amount` NUMERIC(20,4) — **знаковая**: `< 0` отток, `> 0` приток
- `currency` str(3) — денормализовано со счёта на момент операции
- `occurred_at` date
- `merchant` str(300) (nullable)
- `note` str(1000) (nullable)
- `source` str(20) — manual / import / telegram / recurring; в 1.2a только manual
- `transfer_group_id` uuid (nullable) — общий для двух строк перевода
- `created_by` uuid FK → users
- `created_at` timestamptz

Индексы: `(workspace_id, occurred_at)`, `(account_id, occurred_at)`.

### Семантика знака и агрегатов

- Расход: `amount < 0`, `category.kind = expense`.
- Доход: `amount > 0`, `category.kind = income`.
- Перевод: две строки с общим `transfer_group_id`, `category_id = null`;
  одна `amount < 0` (со счёта-источника), вторая `amount > 0` (на счёт-
  получатель). Кросс-валютный перевод отражается естественно (−50000 KRW /
  +3700 RUB — суммы независимы).
- Остаток счёта: `SUM(amount) WHERE account_id = X` (в валюте счёта).
- Расходы месяца по категориям: `SUM(-amount)` группировкой по `category_id`,
  где `amount < 0 AND transfer_group_id IS NULL AND occurred_at` в периоде.
- Переводы всегда исключаются из доход/расход-аналитики по
  `transfer_group_id IS NOT NULL`.

## 4. API

Модуль `ledger`, роутер под `/api`. Все эндпоинты требуют
`get_current_user`; доступ к данным workspace проверяется через membership
(зависимость `require_workspace_member(workspace_id, user)`), 403 если не
участник. `workspace_id` — обязательный query-параметр для доменных данных.

### Счета

- `GET /api/accounts?workspace_id=` → список счетов с полем `balance`
  (строка Decimal).
- `POST /api/accounts` `{workspace_id, name, type, currency}` → 201.
- `PATCH /api/accounts/{id}` `{name?, is_archived?}` → 200.

Удаления счёта нет — только архивация (не осиротить транзакции).

### Категории

- `GET /api/categories?workspace_id=` → дерево (плоский список с parent_id,
  фронт строит дерево).
- `POST /api/categories` `{workspace_id, name, kind, parent_id?}` → 201.
- `PATCH /api/categories/{id}` `{name?, parent_id?}` → 200.

### Транзакции

- `GET /api/transactions?workspace_id=&account_id=&category_id=&from=&to=&limit=&offset=`
  → лента с фильтрами и пагинацией (сортировка по occurred_at desc, id desc);
  ответ включает `total` для пагинации.
- `POST /api/transactions`
  `{workspace_id, account_id, category_id, amount, occurred_at, merchant?, note?}`
  → 201. `amount` — знаковая строка; знак должен соответствовать
  `category.kind` (расход отрицательный, доход положительный) — иначе 422.
  `currency` берётся со счёта.
- `POST /api/transactions/transfer`
  `{workspace_id, from_account_id, to_account_id, from_amount, to_amount, occurred_at, note?}`
  → 201. Создаёт пару атомарно (одна транзакция БД) с общим
  `transfer_group_id`, `category_id = null`, `source = manual`.
  `from_amount` записывается как отрицательная, `to_amount` — положительная.
  Валидация: счета разные, оба в этом workspace.
- `PATCH /api/transactions/{id}` — правка обычной транзакции (не перевода);
  правка строки перевода запрещена (409), редактируется удалением+созданием.
- `DELETE /api/transactions/{id}` → 204. Если у транзакции есть
  `transfer_group_id` — удаляются обе строки группы атомарно.

### Дашборд

- `GET /api/dashboard?workspace_id=` → один объект:
  - `accounts`: `[{id, name, currency, balance}]`
  - `month_expenses`: `[{category_id, category_name, total}]` за текущий
    месяц, сортировка по total desc
  - `recent`: последние 10 операций `[{id, occurred_at, amount, currency,
    account_name, category_name?, merchant?, is_transfer}]`

  Всё считается на сервере; фронт не агрегирует.

## 5. Границы модулей и repository-слой

- Новый модуль `app/ledger/`: `models.py`, `schemas.py`, `repository.py`
  (все запросы + обязательный фильтр workspace_id), `service.py` (доменная
  логика: агрегаты, атомарность перевода, seed категорий), `deps.py`
  (`require_workspace_member`), `router.py`.
- Общение с `identity` — только через сервисный интерфейс: проверка
  членства идёт через публичную зависимость, не запросом в таблицы identity.
- Seed дефолтных категорий при регистрации решается так, чтобы граница
  identity→ledger не возникала вовсе: `register_user` НЕ вызывает ledger.
  Вместо этого роутер регистрации (`identity/router.py`, слой композиции)
  после успешного `register_user` вызывает `ledger.service.seed_categories(
  db, workspace_id)`. Роутеры — слой оркестрации, им разрешено знать оба
  сервиса; сервисы друг друга не импортируют. import-linter контролирует
  именно service/repository-слои, роутеры-композиторы вне контракта
  «forbidden».
- **import-linter** в CI: контракт «модули identity и ledger не лезут во
  внутренности друг друга; core — общий слой». Первый явный барьер границ.
- **repository-слой** — единственное место с SQL; каждый метод принимает
  workspace_id и фильтрует по нему. Прямые запросы в router/service мимо
  repository запрещены.
- **structlog**: middleware/зависимость кладёт `workspace_id` (и `user_id`)
  в contextvars, чтобы доменные логи несли идентификаторы (без PII и сумм).

## 6. Frontend

Поверх паттернов identity (api-клиент, TanStack Query, Mantine, AuthGuard).

- **AppShell** (Mantine): боковая навигация Дашборд / Счета / Категории /
  Операции; шапка — email + выход. Активный workspace в Zustand-сторе
  (пока один, структура готова к переключателю).
- **Дашборд**: карточки счетов с остатками; расходы месяца по категориям
  (список сумм + горизонтальный бар на Mantine `Progress`, без графических
  библиотек); лента последних операций.
- **Счета**: список + модалка создания/правки, архивация.
- **Категории**: дерево с группировкой income/expense, создание/правка.
- **Операции**: таблица с фильтрами (счёт, категория, период), пагинация,
  кнопки «Добавить расход/доход» и «Перевод» (разные формы), правка/удаление.
- Суммы — строками с бэка; формат `Intl.NumberFormat` с валютой счёта;
  `float` для денег не используется нигде.
- Роуты защищены `AuthGuard`; из ревью 1.1 переносится: `AuthGuard`
  различает 401 (→ login) и сеть/500 (→ сообщение об ошибке, не редирект).

## 7. Тестирование

- Backend (pytest + testcontainers postgres/redis):
  - доменная математика: остаток счёта, агрегаты дашборда, знак суммы,
    атомарность перевода (обе строки или ни одной), удаление перевода;
  - **изоляция workspace** — критический тест: пользователь workspace B
    не видит и не может изменить/удалить данные workspace A (404/403);
  - валидация: знак amount vs kind категории (422), перевод между разными
    счетами, счета одного workspace;
  - CRUD-эндпоинты счетов/категорий/транзакций, фильтры и пагинация ленты;
  - seed дефолтных категорий при регистрации.
- Frontend (vitest): формы транзакции и перевода (валидация, отправка),
  рендер дашборда с замоканным API.
- import-linter прогоняется в CI (отдельная проверка или в backend-джобе).

## 8. Принятые развилки

- Знаковый `amount` (не «положительный + флаг направления»): упрощает
  агрегаты и честно отражает переводы.
- Перевод — пара транзакций с `transfer_group_id`, отдельный эндпоинт
  `/transactions/transfer` (не «умный» POST).
- `workspace_id` — query-параметр для доменных данных (не в пути); доступ
  через membership.
- Счета архивируются, не удаляются.
- Дашборд — базовый (остатки + расходы месяца по категориям + лента),
  один серверный агрегирующий эндпоинт; графики/тренды отложены.
- external_id/import_id и дедупликация — часть B (импорт), не в схеме 1.2a.
- Мультивалютная конвертация не вводится; аналитика раздельно по валютам.

## 9. Открытые вопросы

- Реальная выписка Т-Банка нужна для части B (импорт), не для 1.2a.
- Переключатель активного workspace в UI — структура готова (Zustand),
  сам переключатель появится, когда приглашения станут реально
  использоваться (не блокирует 1.2a с одним workspace).
